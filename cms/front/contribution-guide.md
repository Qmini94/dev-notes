# 기여 가이드

BMad 프로젝트에 기여해 주셔서 감사합니다.  
우리는 **대체(Replacement)가 아닌 증폭(Amplification)** 을 지향합니다.  
즉, 인간과 AI가 함께 더 나은 사고를 하도록 돕는 것을 목표로 합니다.

본 가이드는 프로젝트에 기여하는 모든 사람이  
**동일한 철학과 기준을 공유**하도록 하기 위해 작성되었습니다.

---

## 커뮤니티 참여

실시간 논의와 협업은 Discord를 중심으로 이루어집니다.

- **#bmad-development**  
  기술적 논의 및 개발 관련 질문
- **#suggestions-feedback**  
  기능 제안 및 아이디어 공유
- **#report-bugs-and-issues**  
  버그 리포트 및 이슈 논의

---

## 프로젝트 철학

### Human Amplification, Not Replacement

BMad는 인간을 대체하는 도구가 아니라  
**인간의 사고를 증폭하는 프레임워크**입니다.

AI 에이전트는:
- 자동화 도구 ❌
- 결정 대행자 ❌
- **코치, 멘토, 협업자 ⭕**

로 작동해야 합니다.

---

## 좋은 기여란 무엇인가

모든 기여는 다음 질문에 답할 수 있어야 합니다.

> “이 변경이 인간과 AI가 **함께 더 잘 사고**하도록 돕는가?”

### 환영되는 기여

- 인간–AI 협업 패턴을 강화하는 개선
- 에이전트 역할(Persona) 및 워크플로우 개선
- 계획 수립과 컨텍스트 연속성 강화
- 도메인별 BMad 모듈 확장

### 지양해야 할 기여

- 인간 개입을 제거하는 순수 자동화 기능
- 협업을 강화하지 않는 도구성 기능
- 진입 장벽만 높이는 과도한 복잡성
- BMad Core의 일관성을 해치는 변경

---

## 기여 전 준비 사항

### 버그 리포트

- 기존 이슈 중복 여부 확인
- 가능하면 Discord에서 사전 논의
- 버그 리포트 템플릿 사용 권장
  - 재현 방법
  - 기대 결과 vs 실제 결과
  - 환경 정보(BMad 버전, 모델, IDE 등)

---

### 기능 제안

- Discord (#suggestions-feedback)에서 사전 논의 필수
- 기존 이슈/토론 중복 확인
- 기능 요청 템플릿 사용
- **왜 이 기능이 협업을 강화하는지** 명확히 설명

---

## 작업 시작 전 필수 사항

⚠️ PR 제출 전 반드시 확인해야 합니다.

- 버그 수정: 관련 이슈 존재 여부 확인
- 기능 추가: Discord 논의 + 이슈 생성 필수
- 대규모 변경: 반드시 사전 논의

큰 변경은 **작게 쪼개서** 제안해야 합니다.  
정렬되지 않은 대형 PR은 병합되지 않습니다.

---

## Pull Request 가이드

### 브랜치 규칙

다음 경우에만 `main` 브랜치로 PR을 제출합니다.

- 치명적인 기능 장애 수정
- 보안 패치
- 심각하게 잘못된 문서 수정
- 설치 또는 기본 사용을 막는 버그

---

### PR 크기 기준

- 권장: 200–400줄
- 최대: 800줄 (자동 생성 파일 제외)
- PR 하나 = 기능/이슈 하나

큰 변경은 여러 PR로 분리합니다.

---

### 커밋 메시지 규칙

Conventional Commits 형식을 사용합니다.

- `feat:` 새로운 기능
- `fix:` 버그 수정
- `docs:` 문서 수정
- `refactor:` 리팩터링
- `test:` 테스트 추가
- `chore:` 기타 작업

---

## 좋은 PR vs 나쁜 PR

### 좋은 PR

- 하나의 변경만 포함
- 명확한 제목과 설명
- 왜 변경했는지가 분명함
- 관련 이슈 참조 포함

### 피해야 할 PR

- 파일 전체 포맷 변경
- 관련 없는 변경 포함
- 설명 없는 코드 변경
- main 브랜치 직접 작업

## 코드 및 에이전트 작성 가이드

- 개발 에이전트는 **간결하게**

- Core 프레임워크에는 코드 포함 ❌
- 모든 내용은 자연어(Markdown) 기반
