이 문서는 프로젝트의 **디렉토리 구조와 각 영역의 책임**을 설명합니다.  
코드를 어디에 두어야 하는지, 각 폴더가 어떤 역할을 하는지 명확히 하여  
구조적 일관성을 유지하는 것을 목표로 합니다.

본 문서는 신규 참여자와 AI 에이전트 모두가  
**동일한 기준으로 소스 트리를 해석**하도록 돕습니다.

---

## 최상위 구조

프로젝트의 최상위 디렉토리는 다음과 같은 책임 단위로 구성됩니다.

/
├─ pages/
├─ components/
├─ layouts/
├─ middleware/
├─ stores/
├─ composables/
├─ server/
├─ types/
├─ utils/
└─ assets/

각 디렉토리는 명확한 목적을 가지며,  
역할이 겹치지 않도록 설계되었습니다.

---

## pages/

### 역할
- 라우팅 단위의 페이지 정의
- 사용자 접근 경로의 진입점

### 규칙
- 페이지 단위의 조합만 담당
- 복잡한 로직 포함 금지
- 데이터는 store 또는 composables를 통해 주입

---

## components/

### 역할
- 재사용 가능한 UI 컴포넌트 정의

### 규칙
- 상태를 직접 소유하지 않음
- props / emits 기반 통신
- 특정 페이지에 종속되지 않도록 설계

---

## layouts/

### 역할
- 페이지 공통 레이아웃 정의
- 헤더, 푸터, 네비게이션 등 공통 UI 구성

---

## middleware/

### 역할
- 라우트 진입 전 처리 로직
- 인증, 권한 검사, 접근 제어

### 특징
- 전역(global) 미들웨어 우선 사용
- 순서가 의미를 가지므로 파일명에 번호를 포함

### 예시
- `02.authentication.global.ts`
- `04.permission-check.global.ts`

---

## stores/

### 역할
- 전역 상태 관리
- 사용자, 권한, 렌더링 컨텍스트 유지

### 규칙
- 상태 변경은 명시적인 액션을 통해 수행
- UI 컴포넌트에서 직접 변경 금지

---

## composables/

### 역할
- 도메인 단위의 비즈니스 로직 캡슐화
- 재사용 가능한 로직 제공

### 규칙
- 프레임워크 의존 최소화
- UI와 분리된 순수 로직 지향

---

## server/

### 역할
- 서버 사이드 로직
- API 프록시, SSR 관련 처리

---

## types/

### 역할
- 전역 타입 정의
- API, 스토어, 도메인 모델의 타입 명세

### 규칙
- 도메인 기준으로 파일 분리
- any 사용 최소화

---

## utils/

### 역할
- 범용 유틸리티 함수 모음

### 규칙
- 도메인 지식 포함 금지
- 순수 함수 위주 구성

---

## assets/

### 역할
- 정적 리소스 관리
- 이미지, 폰트, 스타일 리소스 포함

---

## 구조 설계 원칙

- 한 디렉토리 = 하나의 책임
- UI ↔ 로직 ↔ 데이터 명확히 분리
- 파일 위치만 보고도 역할이 추론 가능해야 함

이 원칙을 위반할 경우:
- 유지보수 비용 증가
- 코드 탐색 난이도 상승
- AI 기반 자동화 품질 저하
