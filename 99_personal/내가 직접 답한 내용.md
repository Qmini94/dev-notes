## Q1. Elasticsearch 도입 배경

**질문**: Elasticsearch를 기존 시스템에 통합 검색으로 도입하셨다고 했는데, 기존에는 검색을 어떻게 처리하고 계셨고, Elasticsearch를 선택한 이유가 뭔가요?

### 핵심 답변 (30초)

기존 레거시 솔루션은 검색엔진이 따로 없어서 LIKE 기반 DB 직접 조회나 외부 솔루션(와이즈넛)에 의존했습니다. CMS 특성상 동적 게시판 구조여서 하나의 테이블에 모든 게시판 데이터가 들어가 있었고, 성능이 많이 떨어졌습니다. 이를 해결하기 위해 JDBC로 런타임에 게시판별 테이블을 분리하고, 분리된 수십 개 테이블의 통합검색을 위해 ES를 도입했습니다.

### 꼬리질문 대비 디테일

- **테이블 분리**: JDBC로 런타임에 스키마를 수정해서 각 게시판별 테이블을 동적으로 생성 → 개별 게시판의 조회/검색 성능 개선
- **ES 도입 이유**: 분리된 테이블들을 통합검색하려면 전부 JOIN해야 하는 문제 → ES에서 각 게시판 메타 테이블 기준으로 색인해서 통합검색 처리
- **ES 도입 효과**:
    - 제목/내용 검색 속도 향상
    - 검색 정확도(relevance score) 측정 가능 → 사용자 관점에서 데이터 정제 향상
    - 권한을 key로 역색인 → 비회원/직원 권한별 통합검색도 빨라짐

## Q2. ES와 RDB 데이터 동기화

**질문**: ES에 데이터를 색인할 때 RDB와의 동기화는 어떻게 처리하셨나요? 동기화 과정에서 데이터 정합성 이슈는 없었나요?

### 핵심 답변 (30초)

처음에 비동기 처리를 고려했지만, 현재 트래픽 규모와 팀 상황을 고려해 Outbox 폴링 방식을 선택했습니다. 게시판 write 이벤트 발생 시 Outbox 테이블에 함께 트랜잭션 처리하고, 이벤트가 정기적으로 돌면서 ES에 색인합니다. 추가로 매일 새벽 전체 재색인 배치를 돌려 정합성을 보장합니다.

### 꼬리질문 대비 디테일

- **Outbox 패턴 선택 이유**: 메시지 큐 기반 비동기는 컨슈머 실패 시 재처리 로직이 필요하고 인프라 관리 부담이 큼. 현재 팀 규모와 트래픽을 고려하면 오버엔지니어링이라 판단
- **Outbox 동작 방식**: write 이벤트 → 비즈니스 데이터 + Outbox 테이블에 동시 INSERT (같은 트랜잭션) → 폴링으로 Outbox 데이터를 ES에 색인
- **전체 재색인**: 새벽 배치로 전체 재색인 수행. write/read alias swap 방식으로 사용자 서비스가 끊기지 않도록 처리

## Q3. JWT + Redis 인증 시스템

**질문**: JWT와 Redis를 조합한 인증 시스템을 구현하셨다고 했는데, 일반적인 세션 기반 인증 대신 JWT를 선택한 이유는 뭔가요? Redis는 어떤 역할로 사용하셨나요?

### 핵심 답변 (30초)

기존 레거시에서 세션 라이프사이클 관리 실패로 DB I/O 문제를 겪은 경험이 있어서, 사용자 인증/인가가 DB에 의존하지 않도록 JWT를 도입했습니다. 하지만 슬라이딩 토큰이나 권한 수정 처리 등 완전한 stateless는 현실적으로 어려워서 Redis로 하이브리드 구조를 가져갔습니다.

### 꼬리질문 대비 디테일

- **Redis 화이트리스트 구조**: key = 사용자 계정, value = JWT의 SID(UUID) 배열 → 각 세션을 서버에서 관리 가능
- **슬라이딩 방식**: 매 요청마다 JWT 재발급은 비효율적 → 세션 만료시간만 터치하고, 액세스 토큰 만료 후에만 리프레시 토큰으로 재발급
- **인증 흐름**: 세션 타임 + 리프레시 토큰이 있을 때만 사용 가능
- **장애 대응 (Graceful Degradation)**: Redis 장애 시 → 현재 액세스 토큰의 payload로 인증/인가 처리하되 비회원 권한으로 제한

## Q4. HTTP Stateless의 장점과 상태 관리가 필요한 이유

**질문**: HTTP는 stateless 프로토콜인데, stateless의 장점이 뭐고 그럼에도 상태 관리가 필요한 이유는 뭘까요?

### 핵심 답변 (30초)

Stateless는 서버가 클라이언트 상태를 저장하지 않으니까 서버 확장(스케일 아웃)이 쉽고, 어떤 서버가 요청을 받아도 동일하게 처리할 수 있습니다. 로드밸런싱이 자유롭고 서버 장애 시에도 다른 서버로 보내면 되니까 가용성이 높아집니다. 하지만 로그인, 권한, 장바구니 같은 사용자별 맥락이 있어야 서비스가 동작하기 때문에 상태 관리가 필요합니다.

### 본인 경험과 연결하기

> "제가 JWT를 설계할 때도 이 고민을 했습니다. JWT 자체가 stateless하게 인증 정보를 들고 다니자는 건데, 실제로는 슬라이딩 토큰이나 권한 수정 같은 상태 관리가 필요해서 Redis로 하이브리드 구조를 가져간 것도 같은 맥락입니다."

## Q5. 블루/그린 배포 선택 이유

**질문**: 왜 롤링 배포나 카나리 배포 대신 블루/그린을 선택하셨나요?

### 핵심 답변 (30초)

소규모 팀이었고, SaaS가 아닌 각 클라이언트마다 온프레미스로 개별 솔루션이 동작하는 구조였기 때문에 구현 복잡도가 낮고 문제 시 즉시 롤백 가능한 블루/그린이 가장 현실적이었습니다. 각 클라이언트별 독립된 환경이다 보니 트래픽을 분산하거나 순차 교체할 인스턴스가 여러 개 있는 구조가 아니어서 카나리나 롤링은 적합하지 않았습니다.

### 꼬리질문 대비 디테일

- **각 배포 전략 차이점**:
    - 롤링: 인스턴스를 순차 교체 → 배포 중 구버전/신버전 동시 존재, 호환성 문제 가능
    - 카나리: 일부 트래픽만 신버전으로 → 트래픽 라우팅 제어 + 모니터링 인프라 필요
    - 블루/그린: 별도 환경에 신버전 띄운 뒤 한 번에 전환 → 즉시 롤백 가능, 구현 단순
- **구체적 구현**: Docker 컨테이너 두 개를 포트로 분리하고 Nginx에서 upstream을 스위칭하는 방식

## Q6. SSR+CSR 하이브리드 렌더링

**질문**: 백엔드 개발자로서 왜 프론트엔드 렌더링 방식까지 관여하셨나요? SSR과 CSR을 어떤 기준으로 나누셨나요?

### 핵심 답변 (30초)

소규모 팀에서 레거시와 신규 솔루션 개발이 동시에 이루어져야 해서 프론트 코어 로직도 담당했습니다. CMS 특성상 고객사 대부분이 SEO가 필수 요구사항이었기 때문에 SSR은 필수였고, 모든 페이지를 SSR+CSR 하이브리드로 처리했습니다. SSR에서 가져온 데이터를 CSR에서 재사용하도록 처리해서 첫 화면 로딩 속도를 크게 높이면서도 SPA 경험을 유지했습니다.

### 꼬리질문 대비 디테일

- **데이터 재사용 방식**: 라우터 미들웨어나 컴포저블에서 SSR 데이터를 CSR로 넘겨 재사용 (Nuxt의 useAsyncData 등)
- **성과 수치**: Lighthouse 93점 달성, API 응답시간 65% 감소, 40ms 응답 속도
