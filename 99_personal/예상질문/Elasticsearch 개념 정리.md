## 1. Elasticsearch가 뭔지

DB는 데이터를 저장하고 조회하는 거잖아요. ES도 비슷한데, "검색"에 특화된 저장소예요. DB에서 LIKE '%개발%' 하면 80만 건을 처음부터 끝까지 하나씩 비교하는데, ES는 미리 검색용 사전을 만들어놔서 바로 찾아요.

---

## 2. 인덱스가 뭔지

DB에서 테이블이 있잖아요. ES에서는 그걸 인덱스라고 불러요. 프로젝트에서 `cms_content_v1` 이런 게 인덱스예요. 이 안에 문서(document)들이 들어가는데, DB의 row 하나가 ES의 문서 하나라고 생각하면 돼요.

---

## 3. 색인이 뭔지

색인은 "검색할 수 있게 데이터를 넣는 것"이에요. 책 뒤에 있는 찾아보기를 생각해보세요. "경영 → 15쪽, 42쪽, 78쪽" 이런 거요. ES도 데이터를 넣을 때 이런 찾아보기를 자동으로 만들어요. 이걸 **역인덱스**라고 해요.

예를 들어 게시글 3개를 넣으면:

- 문서1: "윤리경영 보고서를 발간했습니다"
- 문서2: "경영지원팀 채용 공고"
- 문서3: "환경 안전 보고서"

ES가 이걸 쪼개서 역인덱스를 만들어요:

- "윤리" → 문서1
- "경영" → 문서1, 문서2
- "보고서" → 문서1, 문서3
- "채용" → 문서2
- "환경" → 문서3

나중에 "경영"을 검색하면 80만 건을 다 뒤지는 게 아니라, 역인덱스에서 "경영 → 문서1, 문서2" 바로 찾아요. 그래서 40ms에 끝나는 거예요.

---

## 4. ES에 들어가는 데이터 구조

게시글 하나가 ES에 들어갈 때 이런 모양이에요:

json

```json
{
  "documentId": "post_news_4848",
  "contentType": "POST",
  "title": "ESG 윤리경영 선언",
  "body": "당사는 윤리경영을 선언...",
  "menuId": 123,
  "boardName": "새소식",
  "visibility": "PUBLIC",
  "siteIdx": 1,
  "siteName": "홈페이지",
  "createdDate": "2024-07-16T09:45:01"
}
```

이건 ES에 저장된 **원본 문서**예요. DB의 row처럼 그냥 데이터가 들어있는 거예요. 색인(역인덱스)은 ES가 이 데이터를 넣을 때 **자동으로** 만드는 거예요.

검색할 때는 역인덱스에서 빠르게 찾고, 검색 결과를 보여줄 때는 원본 문서에서 제목, 본문, 게시판 이름 같은 걸 가져와요.

---

## 5. Text 타입 vs Keyword 타입

ES에서 필드 타입을 정할 때 Text랑 Keyword가 있어요.

**Text**는 노리 같은 분석기가 쪼개서 역인덱스를 만들어요. title이랑 body가 이거예요. "윤리경영 선언" → "윤리", "경영", "선언" 이렇게 쪼개져요.

**Keyword**는 쪼개지 않고 값 그대로 역인덱스에 넣어요. visibility, contentType, siteName이 이거예요. "PUBLIC"은 그냥 "PUBLIC" 그대로 들어가요.

왜 나누냐면, title은 "윤리"로 검색해도 찾아야 하니까 쪼개야 하고, visibility는 "PUBLIC" 정확히 매칭해야 하니까 쪼개면 안 되는 거예요.

---

## 6. Keyword 타입도 역인덱스가 있다

visibility나 contentType 같은 Keyword 타입 필드도 역인덱스가 있어요. 다만 노리처럼 쪼개지 않고 값 그대로 역인덱스를 만들어요.

```
visibility 역인덱스:
"PUBLIC" → 문서1, 문서3, 문서5 ...  (64만건)
"MEMBER" → 문서2, 문서4 ...        (12만건)
"ADMIN"  → 문서6, 문서8 ...        (4만건)

contentType 역인덱스:
"POST" → 문서1, 문서2, 문서5 ...
"PAGE" → 문서3, 문서7 ...
"MENU" → 문서4, 문서6 ...
```

그래서 filter에 visibility=PUBLIC 넣으면 80만 건을 하나씩 확인하는 게 아니라 역인덱스에서 PUBLIC에 해당하는 문서 목록을 바로 가져와요. **텍스트든 필터든 전부 역인덱스 기반이라 빠른 거예요.**

---

## 7. 권한 처리

검색할 때 모든 게시글을 다 보여주면 안 되잖아요. 비공개 글을 비로그인 사용자가 보면 안 되니까요. 그래서 색인할 때 각 문서에 visibility를 넣어둬요. PUBLIC이면 누구나, MEMBER면 로그인 사용자만, ADMIN이면 관리자만 볼 수 있어요.

검색 요청이 오면 백엔드에서 이 사용자가 누군지 확인하고, ES 쿼리에 필터를 붙여요. 비로그인이면 "visibility가 PUBLIC인 것만", 관리자면 "전부 다", 일반 사용자면 "PUBLIC이랑 MEMBER만, 그중에서도 메뉴 접근 권한 있는 것만" 이렇게요.

경력기술서 기준으로는, 색인 시 각 게시글이 속한 메뉴의 권한을 pathId 기반으로 상속 권한까지 계산해서 ES 문서에 포함시킴. 검색할 때는 ES의 bool 쿼리에서 filter로 사용자 권한에 맞는 문서를 먼저 걸러내고, must에서 노리로 텍스트 매칭을 수행. 권한 필터링과 텍스트 검색이 하나의 ES 쿼리로 처리되는 구조.

---

## 8. bool 쿼리 구조

ES에서 검색할 때 bool 쿼리를 쓰는데, 이게 조건을 조합하는 방식이에요. SQL의 WHERE절이랑 비슷해요.

- **must**: "이 조건에 매칭되어야 하고, 얼마나 잘 매칭되는지 점수도 매긴다." 검색어 매칭을 여기에 넣어요. "경영"이 제목에 있으면 본문에만 있는 것보다 점수가 높아서 위에 나와요.
- **filter**: "이 조건에 맞는 것만 걸러내고, 점수는 안 매긴다." 권한이나 사이트 같은 조건을 여기에 넣어요. PUBLIC이냐 MEMBER냐가 검색 순위에 영향을 줄 이유가 없으니까요. filter는 점수 계산을 안 하니까 must보다 빠르고, ES가 결과를 캐싱해서 같은 필터 조건이 반복되면 더 빨라요.
- **should**: "이 조건에 매칭되면 점수를 더 준다." 필수는 아닌데 가점을 주는 거예요.
- **must_not**: "이 조건에 매칭되면 제외한다." SQL의 NOT이랑 같아요.

실제 프로젝트 쿼리 구조:

```
bool:
  must: "경영" 텍스트 매칭 (제목 가중치 2배)
  filter:
    - visibility = PUBLIC
    - siteName = "홈페이지"
    - contentType = POST
```

이러면 "홈페이지 사이트의 공개 게시글 중에서 '경영'이 포함된 것을 제목 매칭 우선으로 정렬해서 보여줘"가 되는 거예요.

---

## 9. filter 캐싱

filter 조건은 ES가 자동으로 캐싱해요. 예를 들어 첫 번째 사용자가 visibility=PUBLIC으로 검색하면 ES가 "PUBLIC인 문서 목록"을 캐시에 저장해요. 두 번째 사용자가 같은 조건으로 검색하면 다시 계산 안 하고 캐시에서 바로 가져와요. 그래서 반복 검색이 많을수록 빨라지는 거예요.

---

## 10. 제목 가중치 2배 (title^2)

코드에서 multi_match 쿼리에 `title^2, body`라고 설정했어요. 이건 제목에서 매칭되면 점수를 2배로 준다는 뜻이에요.

사용자가 "경영"을 검색했을 때:

- 문서A: 제목에 "경영" 있음 → 점수 20점
- 문서B: 본문에만 "경영" 있음 → 점수 10점

제목에 검색어가 있는 게 더 관련성 높을 가능성이 크니까 위에 보여주는 거예요.

---

## 11. multi_match 쿼리

여러 필드에서 동시에 검색하는 거예요. title이랑 body 두 필드에서 검색하니까 multi_match를 쓰면 한 번의 쿼리로 "제목에서도 찾고 본문에서도 찾아줘"가 되는 거예요. 필드마다 따로 쿼리 안 날려도 돼요.

---

## 12. Nori 분석기

아까 역인덱스를 만들려면 문장을 쪼개야 한다고 했잖아요. 영어는 띄어쓰기로 쪼개면 되는데 한국어는 안 돼요. "윤리경영을 선언했습니다"를 띄어쓰기로 쪼개면 "윤리경영을", "선언했습니다" 이렇게 되는데, 사용자가 "윤리"나 "경영"으로 검색하면 못 찾아요.

Nori는 한국어를 이해해서 "윤리경영을 선언했습니다"를 이렇게 쪼개요:

- "윤리" + "경영" + "선언" + "하" + "습니다"

조사("을")랑 어미("했습니다")를 제거하고 의미 있는 단어만 남기는 거예요. 그래서 "윤리"로 검색해도, "경영"으로 검색해도 다 찾을 수 있어요.

Nori는 ES 공식 플러그인이에요. ES에 기본 내장은 아니고 별도로 설치해야 하는데, Elastic 공식 팀이 만들고 관리하는 거라 서드파티가 아니에요. Dockerfile에서 `bin/elasticsearch-plugin install analysis-nori` 한 줄로 설치함.

---

## 13. 복합명사랑 decompound_mode: mixed

복합명사는 두 개 이상의 단어가 합쳐진 거예요. "삼성전자", "윤리경영", "정보통신" 이런 거요.

decompound_mode에는 세 가지가 있어요:

- **none**: 안 쪼갬. "삼성전자" 그대로 저장. "삼성"으로 검색하면 못 찾음.
- **discard**: 쪼개고 원본 버림. "삼성" + "전자"만 저장. "삼성전자"로 검색하면 못 찾음.
- **mixed**: 쪼개고 원본도 유지. "삼성전자" + "삼성" + "전자" 다 저장.

mixed로 설정한 이유: "삼성전자"로 검색해도 되고, "삼성"만으로 검색해도 되고, "전자"만으로 검색해도 다 매칭돼요. 역인덱스에 세 개가 다 들어가 있으니까요.

**트레이드오프**: mixed는 역인덱스에 저장되는 양이 많아져서 인덱스 크기가 커지고 색인 속도가 약간 느려질 수 있어요. 근데 80만 건 수준에서는 영향 미미하고, 검색 품질이 우선이라 mixed를 선택함.

## 14. 하이라이팅

검색 결과에서 매칭된 부분을 강조 표시하는 거예요. "ESG 윤리경영 선언"에서 "경영"을 검색하면 결과에 "ESG 윤리**경영** 선언" 이렇게 `<strong>` 태그로 감싸서 보여주는 거예요. 응답의 snippet 필드에 `<strong>ESG</strong>` 이렇게 들어가 있는 게 이거예요. ES가 자동으로 해줘요.

---

## 15. Alias (별명)

실제 인덱스 이름은 `cms_content_v1`인데, 여기에 `cms_content_read`라는 별명을 붙여놓는 거예요. 검색 서비스는 항상 `cms_content_read`로 쿼리해요.

재색인할 때 `cms_content_v2`를 새로 만들고, 다 끝나면 `cms_content_read` 별명을 v2로 옮기면 검색 서비스는 코드 변경 없이 자연스럽게 새 인덱스를 보게 돼요. 이게 **Alias Swap 무중단 재색인**이에요.

---

## 16. Outbox 패턴

게시글을 저장할 때 DB에도 넣고 ES에도 넣어야 하잖아요. 근데 DB 저장은 성공했는데 ES 색인이 실패하면? DB에는 있는데 검색이 안 되는 거예요. 반대로 ES에는 들어갔는데 DB가 롤백되면 검색에는 나오는데 실제로는 없는 글이에요.

Outbox 패턴은 게시글 저장할 때 **같은 DB 트랜잭션 안에서** "이 게시글 ES에 넣어야 해"라는 이벤트를 Outbox 테이블에 같이 넣어요. DB가 롤백되면 Outbox도 같이 롤백되니까 일관성이 보장돼요. 별도 프로세스가 Outbox 테이블을 주기적으로 확인해서 미처리 이벤트를 ES에 색인하는 거예요.

**감사 로깅의 Redis Streams와 비교**: 감사 로그는 유실돼도 파일로 폴백하면 되니까 Redis Streams로 충분했지만, 검색 색인은 DB랑 정확히 일치해야 하니까 같은 트랜잭션 안에서 처리하는 Outbox 패턴이 필요했던 거예요.
