## 1. MSA (Microservice Architecture) 핵심

### 1-1. 서비스 간 통신

|방식|설명|장점|단점|
|---|---|---|---|
|**동기 (REST, gRPC)**|A가 B를 HTTP로 호출하고 응답 대기|구현 단순, 즉시 결과 확인|B 죽으면 A도 장애 (강한 결합)|
|**비동기 (Kafka, RabbitMQ)**|A가 메시지를 큐에 넣고, B가 나중에 처리|결합도 낮음, B 죽어도 A 정상|즉시 결과 확인 어려움, 구현 복잡|

> 💡 **면접 TIP:** "언제 동기, 언제 비동기?" → 즉시 응답 필요하면 동기(결제 확인), 처리 지연 허용되면 비동기(알림 발송)

---

### 1-2. 분산 트랜잭션

모놀리식에서는 하나의 DB 트랜잭션으로 끝나지만, MSA에서는 서비스마다 DB가 다르기 때문에 문제 발생.

**SAGA 패턴**

- 각 서비스가 로컬 트랜잭션을 순서대로 실행
- 중간에 실패하면 **보상 트랜잭션**으로 롤백
- 예시: 주문 → 결제 → 재고차감. 재고차감 실패 → 결제 취소(보상) → 주문 취소(보상)

|방식|설명|
|---|---|
|**Choreography**|각 서비스가 이벤트를 듣고 알아서 반응 (분산적)|
|**Orchestration**|중앙 오케스트레이터가 순서 지시 (중앙 관리)|

---

### 1-3. Redis 분산 락 vs SAGA 패턴 (구분 필수!)

||Redis 분산 락|SAGA 패턴|
|---|---|---|
|**해결하는 문제**|동시 접근 충돌|서비스 간 트랜잭션 일관성|
|**범위**|같은 자원에 대한 동시성 제어|서로 다른 서비스/DB 간|
|**예시**|재고 1개에 2명 동시 주문 방지|주문→결제→재고 흐름 관리|

java

```java
// Redis 분산 락 예시 (Redisson)
RLock lock = redisson.getLock("lock:product:123");
lock.lock(3, TimeUnit.SECONDS);
try {
    // 재고 차감 로직
} finally {
    lock.unlock();
}
```

> 💡 **면접 TIP:** "둘 다 MSA에서 중요하지만 해결하는 문제가 다릅니다. Redis 분산 락은 동시성 제어, SAGA는 서비스 간 트랜잭션 일관성을 보장합니다."

---

### 1-4. 장애 전파 방지

|패턴|설명|
|---|---|
|**서킷 브레이커**|B 서비스 호출 실패가 일정 횟수 넘으면 호출 차단. Closed(정상) → Open(차단) → Half-Open(테스트). Java: **Resilience4j**|
|**폴백 (Fallback)**|서킷 열리면 대체 응답 반환 (캐시 데이터, 기본값 등)|
|**타임아웃 + 재시도**|무한 대기 방지, 재시도는 지수 백오프 (1초→2초→4초)|
|**벌크헤드 (Bulkhead)**|서비스별 스레드풀 분리 → 하나가 느려도 다른 서비스 영향 없음|

---

### 1-5. 서비스 디스커버리 & API Gateway

|구성요소|설명|
|---|---|
|**서비스 디스커버리**|MSA에서 인스턴스 IP가 계속 바뀜 → Eureka(Spring Cloud)로 레지스트리에 자동 등록/조회. K8s 환경이면 K8s Service가 대신|
|**API Gateway**|클라이언트 → Gateway → 각 서비스. 인증, 라우팅, 로드밸런싱, 속도제한을 한 곳에서 처리. Spring Cloud Gateway|

## 2. DDD (Domain-Driven Design) 핵심

### 2-1. 바운디드 컨텍스트 (Bounded Context)

같은 단어라도 맥락에 따라 의미가 다르다. 각 컨텍스트가 자기만의 모델을 가지고, MSA에서는 이것이 보통 **서비스 경계**가 된다.

|주문 컨텍스트|재고 컨텍스트|상품 컨텍스트|
|---|---|---|
|"상품" = 주문 항목 (가격, 수량)|"상품" = 창고 물품 (위치, 수량)|"상품" = 카탈로그 정보 (설명, 이미지)|

---

### 2-2. 애그리거트 (Aggregate)

관련 엔티티를 하나로 묶은 단위. **애그리거트 루트**를 통해서만 내부 데이터를 변경한다.

- **예시:** Order(루트) → OrderItem, ShippingInfo
- 외부에서 OrderItem 직접 수정 ❌
- 반드시 Order를 통해서 수정 ✅
- **이유:** 데이터 일관성 보장. 주문 총액 계산 같은 비즈니스 규칙이 루트에서 관리됨

---

### 2-3. 엔티티 vs 값 객체 (Value Object)

||엔티티 (Entity)|값 객체 (Value Object)|
|---|---|---|
|**식별**|고유 ID가 있음|ID 없음, 값 자체가 의미|
|**변경**|상태가 변해도 같은 것|불변 (Immutable)|
|**예시**|회원, 주문|주소, 금액(Money)|

java

```java
// 값 객체 예시
public record Money(int amount, String currency) {
    public Money add(Money other) {
        return new Money(this.amount + other.amount, this.currency);
    }
}
```

---

### 2-4. 도메인 이벤트

상태 변경 시 이벤트를 발행해서 다른 컨텍스트에 알림. MSA에서 서비스 간 **느슨한 결합**을 만드는 핵심 수단.

- 주문 완료 → `OrderCompletedEvent` 발행 → 결제 서비스가 수신해서 처리

---

### 2-5. 헥사고날 아키텍처 (포트와 어댑터)

|핵심|설명|
|---|---|
|**원칙**|도메인 로직이 중심, 외부(DB, API)에 의존하지 않음|
|**구조**|Port(인터페이스)를 통해 외부와 소통|
|**장점**|DB가 바뀌어도 도메인 코드는 안 바뀜|

## 3. 면접 빈출 꼬리질문

**Q. "모놀리식에서 MSA로 전환하려면 어떻게 접근하겠는가?"**

> 스트랭글러 패턴 (점진적 전환). 기존 모놀리식을 유지하면서 새로운 기능을 MSA로 분리해나가는 방식.

**Q. "MSA의 단점은?"**

> 분산 트랜잭션 복잡성, 운영 비용 증가, 디버깅 어려움, 네트워크 레이턴시, 서비스 간 테스트 복잡성.

**Q. "DDD를 실무에서 적용해본 적 있나?"**

> → **내 CMS 프로젝트에서 도메인 분리한 경험과 연결해서 답변!**

**Q. "CQRS가 뭔가?"**

> 명령(쓰기)과 조회(읽기) 모델을 분리. → **내 Elasticsearch가 읽기 모델로 쓰인 것과 연결 가능! DB에 쓰고 ES에서 읽는 구조가 CQRS의 실무 적용 사례.**

**Q. "신기술 도입에 대한 생각은?"**

> 프로덕션에 검증된 기술 우선. 새 기술은 PoC를 통해 검증 후 점진적 도입. 팀의 학습 비용과 운영 부담도 고려해야 한다.

**Q. "문제해결 방법은?"**

> 문제 정의 → 로그/모니터링으로 원인 파악 → 가설 수립 → 검증 → 해결. **실무 예시를 곁들여서 답변하면 좋다.**

## 4. 자료구조 기초 (면접 빈출)

| 자료구조                        | 특징                                                                | 사용 예시                                   |
| --------------------------- | ----------------------------------------------------------------- | --------------------------------------- |
| **HashMap**                 | Key-Value 저장, O(1) 조회. 해시 충돌 시 O(n)                               | 캐시, 세션 관리                               |
| **ArrayList vs LinkedList** | ArrayList: 인덱스 조회 O(1), 삽입 O(n) / LinkedList: 조회 O(n), 삽입/삭제 O(1) | ArrayList: 읽기 위주 / LinkedList: 삽입/삭제 빈번 |
| **Stack / Queue**           | Stack: LIFO / Queue: FIFO                                         | Stack: 실행취소, 괄호검증 / Queue: 작업 대기열, BFS  |
| **Tree / B-Tree**           | B-Tree: DB 인덱스에 사용. 균형 잡힌 트리로 O(log n) 보장                         | DB 인덱스 (MySQL InnoDB)                   |
