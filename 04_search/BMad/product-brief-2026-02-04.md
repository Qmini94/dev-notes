# Product Brief

## Executive Summary

CMS 통합 검색 시스템은 멀티사이트 구조에서 분산된 콘텐츠를 단일 검색 경험으로 통합하는 프로젝트이다. 현재 CMS는 게시판 단위의 단순 키워드 검색만 제공하며, 사이트 간·도메인 간 통합 검색이 불가능하다. 이로 인해 시민(최종 사용자)과 관리자 모두 필요한 정보를 찾기 위해 메뉴 구조를 직접 탐색해야 하는 비효율이 존재한다.

본 프로젝트는 Elasticsearch 기반 하이브리드 검색(BM25 키워드 + 벡터 kNN 시맨틱) 구조를 도입하여, 키워드 매칭을 넘어 의미 기반 콘텐츠 탐색까지 지원한다. 게시판 게시글, 페이지 콘텐츠, 메뉴 정보, 첨부파일 메타데이터를 통합 인덱싱하며, 사용자 권한과 사이트 정책에 따라 접근 불가 콘텐츠는 검색 결과에서 완전히 제외한다.

이를 통해 CMS의 정보 탐색 패러다임을 "메뉴 탐색 방식"에서 "검색 중심 방식"으로 전환하는 것을 목표로 한다.

## Core Vision

### Problem Statement

현재 CMS는 멀티사이트 + 메뉴 기반 구조로 콘텐츠가 여러 도메인에 분산되어 있다. 사용자가 특정 정보를 찾으려면 사이트를 개별 이동하거나 메뉴 구조를 직접 따라가야 하며, CMS 전체를 아우르는 통합 검색 기능이 존재하지 않는다. 게시판 단위의 단순 키워드 검색만 가능한 상태로, 사이트 간 콘텐츠 통합 검색, 권한 기반 결과 필터링, 의미 기반 탐색이 모두 불가능하다.

핵심 문제: **"정보는 존재하지만 찾기 어렵다."**

### Problem Impact

- **시민/최종 사용자**: 필요한 행정 정보, 공지, 안내 콘텐츠를 빠르게 찾지 못하며, 정확한 메뉴 위치를 알지 못하면 정보 접근 자체가 어려움
- **관리자/내부 사용자**: 과거 게시물이나 특정 콘텐츠를 찾는 데 과도한 시간이 소요되며, 콘텐츠가 증가할수록 관리 효율이 지속적으로 저하됨
- **시스템 전체**: 콘텐츠 자산의 활용도가 낮아지고, 정보 접근성 부족으로 CMS의 본래 가치가 훼손됨

### Why Existing Solutions Fall Short

- 게시판별 단순 키워드 매칭만 가능하여 CMS 전체를 아우르는 검색 불가
- 사이트 간, 도메인 간 콘텐츠를 한 번에 탐색할 수 없음
- 권한 및 공개 여부에 따른 검색 결과 제어가 어려움
- 유사 의미 콘텐츠 탐색이 불가능하여 정확한 키워드를 모르면 정보를 찾을 수 없음
- 콘텐츠 유형(게시글, 페이지, 메뉴, 파일)이 분리되어 있어 통합 탐색 불가

### Proposed Solution

Elasticsearch(또는 OpenSearch) 기반 단일 검색 엔진 구조로 CMS 전체 콘텐츠를 통합 인덱싱하고, 하이브리드 검색을 제공한다.

**검색 대상:** 게시판 게시글, 페이지 콘텐츠, 메뉴명/메뉴 경로, 첨부파일 메타데이터 (첨부파일 내부 텍스트는 추후 확장 검토)

**검색 방식:**
- BM25 기반 키워드 검색 (정확한 텍스트 매칭)
- 벡터 필드 + kNN 기반 시맨틱 검색 (의미 유사도 탐색)
- 두 방식을 결합한 Hybrid Search 구조

**검색 범위 제어:**
- 기본: 전체 사이트 대상 검색
- 필터: 특정 사이트, 콘텐츠 유형별(게시판, 페이지 등) 범위 선택
- 추후 게시판 단위 필터 확장 가능

**권한 기반 필터링:**
- 비공개 콘텐츠는 검색 결과에서 완전히 제외
- 사이트 정책, 사용자 권한 수준, 공개/비공개 상태를 반영
- 검색 결과는 실제 접근 가능한 정보 범위와 100% 일치

### Key Differentiators

- **단일 검색 엔진 구조**: 별도 벡터 DB 없이 Elasticsearch 내에서 키워드+시맨틱 검색을 모두 처리하여 운영 복잡도를 최소화
- **멀티사이트 통합 검색**: 분산된 도메인별 콘텐츠를 검색 레벨에서 통합하여 단일 검색 경험 제공
- **권한-검색 완전 동기화**: 접근 불가 콘텐츠의 존재 자체를 노출하지 않는 엄격한 보안 정책 적용
- **하이브리드 검색**: 키워드 정확도와 의미 유사도를 결합하여 사용자 의도에 맞는 결과 제공
- **검색 중심 패러다임 전환**: CMS의 정보 탐색 방식을 메뉴 탐색에서 검색 중심으로 전환

## Target Users

### Primary Users

#### 1. 시민 사용자 (사이트 방문자)

**프로필:** 행정 정보, 민원 절차, 공지사항, 자료 등을 찾기 위해 CMS를 방문하는 일반 시민

**대표 유형:**
- **민원 탐색자**: "신청 방법", "제출 서류", "지원 대상" 등 특정 행정 절차 정보를 검색
- **공지 확인자**: 정책 변경, 일정, 모집 공고 등 최신 안내를 확인
- **자료 이용자**: 신청서 양식, 매뉴얼, 안내 자료 등을 다운로드하거나 참고

**핵심 특징:** 정확한 메뉴 구조를 알지 못하며, 검색을 통해 바로 원하는 정보에 도달하는 것이 핵심 요구사항

**대표 시나리오:** 사용자가 키워드를 입력하면 CMS 전체에서 관련 정보를 검색하고, 메뉴 경로를 몰라도 필요한 콘텐츠에 직접 접근한다.

#### 2. 콘텐츠 작성자

**프로필:** 게시글, 페이지 콘텐츠를 작성 및 수정하는 내부 담당자

**핵심 니즈:** 과거 작성 콘텐츠나 유사 사례를 찾기 위해 검색을 가장 빈번하게 사용. 콘텐츠가 누적될수록 과거 자료를 찾는 시간이 증가함.

**대표 시나리오:** 특정 키워드로 기존 게시물을 찾아 참고하거나 수정하며, 사이트 단위 또는 도메인 단위로 범위를 좁혀 검색한다.

#### 3. 사이트 관리자

**프로필:** 특정 사이트의 콘텐츠를 관리하는 담당자

**핵심 니즈:** 게시물 상태나 위치 확인, 콘텐츠 관리 효율을 위해 검색 필요. 콘텐츠 작성자와 함께 검색을 가장 많이 필요로 하는 역할.

### Secondary Users

#### 시스템 관리자

**프로필:** 전체 CMS를 운영하는 최상위 관리자

**핵심 니즈:** 특정 콘텐츠 존재 여부 확인, 이슈 대응, 데이터 점검 시 검색 활용

#### 타 부서 직원 / 내부 협업 사용자

**프로필:** CMS에 게시된 정보를 업무 참고용으로 활용하는 내부 사용자

#### 외부 시스템 연동 (추후 확장)

**프로필:** API 기반 검색 기능을 통해 다른 시스템에서 CMS 정보를 조회하는 외부 연동 대상. 1차 대상은 아니지만, 통합 검색 인프라 구축 시 추가 활용 가능.

### User Journey

#### 시민 사용자 여정

1. **진입**: CMS 사이트 방문 → 검색창 발견
2. **검색**: 키워드 입력 (예: "신청 방법") → 전체 사이트 통합 결과 확인
3. **탐색**: 결과 목록에서 콘텐츠 유형/사이트별 필터 적용 → 관련 결과 좁히기
4. **도달**: 원하는 콘텐츠 클릭 → 해당 페이지로 직접 이동
5. **가치 실현**: 메뉴 구조를 몰라도 필요한 정보에 빠르게 접근 완료

#### 관리자 여정

1. **필요 발생**: 과거 게시물 참고, 콘텐츠 확인, 이슈 대응 등
2. **검색**: 관리 화면에서 키워드 입력 → 사이트/도메인 단위로 범위 설정
3. **결과 확인**: 콘텐츠 유형별 필터로 게시글/페이지 등 구분하여 탐색
4. **작업 수행**: 해당 콘텐츠로 이동하여 수정, 참고, 상태 확인
5. **효율 향상**: 누적된 콘텐츠 속에서도 빠른 검색으로 관리 시간 절감

## Success Metrics

### 사용자 성공 지표

- **상위 결과 정확도**: 사용자가 찾는 콘텐츠가 상위 3~5개 결과 안에 포함되는 비율
- **클릭 후 도달 성공률**: 검색 결과 클릭 후 추가 탐색 없이 목적 콘텐츠에 도달하는 비율
- **검색 응답 속도 체감**: 검색 결과가 즉시 표시된다고 느낄 수준의 속도 (체감 지연 거의 없음)
- **탐색 방식 변화**: 메뉴 탐색 대비 검색을 주요 탐색 수단으로 사용하는 비율 증가

### Business Objectives

**3개월 이내:**
- 메뉴 기반 탐색 대비 검색 사용 비율 증가
- 관리자 콘텐츠 검색 및 관리 작업 시간 단축
- 검색 사용 로그 수집 체계 확립

**12개월 이내:**
- 시민 사용자의 정보 탐색 실패율 감소
- 반복 문의 또는 유사 민원 감소
- 콘텐츠 재사용 및 참조율 증가
- 검색 로그 기반 개선 체계 정착

**목표:** 검색이 CMS의 보조 기능이 아니라 주요 정보 탐색 수단으로 자리 잡는 것

### Key Performance Indicators

**기술적 성능 지표:**

| 지표 | 목표 |
|------|------|
| 평균 검색 응답 시간 | 300ms 이하 |
| 최대 검색 응답 시간 | 1초 이하 |
| 색인 반영 지연 | 콘텐츠 변경 후 수 분 이내 |
| 검색 커버리지 | 검색 대상 콘텐츠 거의 전량 인덱스 포함 |
| 권한 정확도 | 접근 불가 콘텐츠 노출 차단 100% |

**시스템 설계 및 운영 지표:**

- **모듈 독립성**: CMS와 연동되지만 독립 모듈 구조, 다른 프로젝트에서 재사용 가능한 형태
- **낮은 시스템 종속성**: 데이터 매핑 계층만 교체하면 다른 시스템에도 적용 가능
- **운영 편의성**: 색인 상태, 검색 성능, 에러 로그 모니터링 가능, 검색 시스템 독립 관리
- **확장 가능성**: 도메인 추가, 검색 대상 확장, 벡터 검색 고도화가 구조 변경 없이 점진적으로 가능
- **장애 격리**: 검색 시스템 장애가 CMS 전체 장애로 확산되지 않는 격리 구조
- **유지보수 용이성**: 색인 파이프라인, 검색 API, 권한 필터링 로직이 명확히 분리되어 수정 시 영향 범위 최소화

## MVP Scope

### Core Features

**목적:** CMS 전역에서 동작하는 안정적인 통합 검색 기반 구축. 기능의 수보다 검색 시스템의 구조적 기반 완성에 집중.

**1. 키워드 기반 통합 검색**
- BM25 기반 텍스트 검색으로 CMS 전역 콘텐츠 대상 검색
- 검색 대상: 게시판 게시글, 페이지 콘텐츠, 메뉴명 및 메뉴 경로 정보
- 정렬: 정확도순 기본, 최신순 정렬 옵션 제공

**2. 권한 기반 필터링 (필수)**
- 사용자 권한, 사이트 정책, 공개/비공개 상태에 따른 결과 필터링
- 접근 불가 콘텐츠는 검색 결과에서 완전히 제외
- 검색 결과는 실제 접근 가능한 정보 범위와 100% 일치

**3. 검색 범위 제어**
- 기본: 전체 사이트 대상 검색
- 필터: 특정 사이트 단위 범위 축소, 콘텐츠 유형별(게시판, 페이지 등) 필터링

**4. 확장 가능한 검색 아키텍처**
- 검색 데이터 매핑 계층을 별도로 두어 CMS 도메인 로직과 직접 결합 방지
- 새로운 도메인(조직 정보, 회원 정보 등) 추가 시 데이터→검색 문서 변환 매핑만 추가하면 되는 구조
- 공통 인덱스 모델: title, content, type, site, accessLevel 등 공통 필드 중심 설계
- 게시판, 페이지, 조직, 회원 등은 "콘텐츠 타입"으로 구분
- 특정 테이블 구조에 강하게 종속되지 않는 범용 설계 (다른 프로젝트/시스템 재사용 가능)

### Out of Scope for MVP

- **벡터 기반 시맨틱 검색**: 고도화 단계에서 도입
- **첨부파일 내부 텍스트 색인**: 파일명 등 메타데이터만 검색 대상으로 한정
- **검색 로그 분석 대시보드**: 후속 단계로 이관
- **외부 시스템 API 기반 검색 연동**: 후속 단계로 이관
- **관리자 검색 대상 도메인 설정 UI**: 아키텍처적으로는 확장 가능하나 MVP에서 UI 미제공

### MVP Success Criteria

- 사용자가 CMS 전역에서 키워드 기반으로 원하는 콘텐츠를 찾을 수 있음
- 접근 권한이 없는 콘텐츠가 검색 결과에 노출되지 않음
- 평균 검색 응답 속도가 체감상 즉시 수준
- 관리자 및 시민 사용자 모두 기존 메뉴 탐색 대비 검색 사용 비율 증가
- 검색 기능이 안정적으로 운영되며 색인 누락이나 권한 오류가 없는 상태 유지

### Future Vision

**고도화 1단계:**
- 벡터 필드 + kNN 기반 시맨틱 검색 도입 (Elasticsearch 내 하이브리드 검색)
- 첨부파일 내부 텍스트 색인 지원
- 검색 로그 수집 및 분석 대시보드

**고도화 2단계:**
- 외부 시스템 API 기반 검색 연동
- 검색 대상 도메인 관리 UI (관리자가 직접 검색 대상 설정)
- 게시판 단위 필터 확장
- 검색 로그 기반 자동 개선 체계 (인기 검색어, 검색 실패 분석 등)

**장기 비전:**
- 검색이 CMS의 핵심 정보 탐색 수단으로 자리 잡음
- 조직 정보, 회원 정보 등 비콘텐츠 도메인까지 검색 범위 확장
- 다른 전자정부 프로젝트에서 검색 모듈 재사용
